emdesign AI-Native Design IDE

Proposal · Design · Specs · Tasks — openspec/changes/ai-native-design-ide
🎯 Evolve the addon from separate tools (comment, live-edit, chat) into a conversation-first design engineering IDE — like Cursor for design, running on Storybook.

Why

Before

  • Comment tool isolated from chat
  • Live-edit has no AI context
  • Chat knows nothing about the view
  • No per-story conversations
  • AI works blind — no component metadata
  • Tabbed panels (Emdesign | Charters | Chat)

After

  • Chat is the primary interface
  • AI sees viewport, component, file, tokens
  • Conversations scoped per story or global
  • Every component is self-describing to AI
  • Natural language edits with auto-verification
  • IDE layout: chat column + tool output

Key Design Decisions

Context flows from preview iframe → channel → manager → backend

The preview iframe has direct DOM access. We extend the existing channel protocol with EVT_VIEW_CONTEXT rather than having the backend query Storybook indirectly.

Component harness as a Storybook decorator

Zero migration cost — add one decorator to preview.ts and every component is wrapped with metadata, error boundary, and AI-accessible attributes.

Conversation scope derived from story ID

Story ID is a stable, unique key. URL-driven switching means no complex state management — navigating to a story loads its conversation.

Design Surface API as a cached query endpoint

The AI needs a programmatic way to query surface state. 5-second cache prevents redundant evaluations while staying fresh enough for interactive use.
